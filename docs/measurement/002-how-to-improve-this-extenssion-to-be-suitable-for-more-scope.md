**Metric 2: MemPalace call silence ratio (unconfigured path)**
- **What it measures:** Number of `mempalace_` substrings present in any instruction string generated by the extension when `mempalaceUrl` is not set in config; additionally, `bun run smoke:no-memory` exit code
- **Why it matters:** Goal 2 тАФ zero MemPalace calls without `mempalaceUrl`. Currently `buildEvaluateInstruction()` unconditionally inserts a step calling `mempalace_add_drawer`; the memory scout is unconditionally included in the 4-scout default roster in `runScout()`; both must be gated
- **Predicted baseline:** 2 unconfigured emission sites тАФ (a) `buildEvaluateInstruction()` contains `mempalace_add_drawer` regardless of config; (b) memory scout called unconditionally in `runScout()` with no `mempalaceUrl` guard
- **Predicted after change (Layer 2, Bugs E + I):** 0 тАФ `buildEvaluateInstruction` omits MemPalace step when `config.mempalaceUrl` is falsy; `runScout()` skips the memory scout when `mempalaceUrl` is not set and uses default roster `[web, oss, repo]`
- **Confidence in prediction:** High for the unit test assertion (string content is deterministic); Medium for the smoke test (depends on the smoke harness being written as part of Layer 3)
- **How to measure:**


## Sample size rationale

This is a developer-tool extension with no production traffic. The verification approach is therefore deterministic, not statistical:

**Deterministic code-search assertions (Metrics 1, 3, 4 in part):** Grep results and test pass/fail are binary and deterministic. A single CI run is sufficient тАФ the "sample" is the entire codebase, inspected in full.

**Deterministic unit tests (Metrics 2, 3, 4):** Each test exercises one control path with a fixed input. The 97-test baseline already proves this model works. Adding 12 new tests (7 bug-fix + тЙе5 plugin contract) keeps the suite well within Vitest's sub-second per-test performance profile; flakiness from resource contention is not a concern.

**Smoke tests (Metrics 2, 5):** Scripted against fixed env-var states. They assert that three specific configurations produce three specific exit codes. Three scenarios cover the full decision tree (ok / warning / error). No larger sample is needed.

**Secondary signal (not a primary metric):** A volunteer (not the author) should be able to complete `/research:new` тЖТ `/research:scout` on a fresh clone in under 15 minutes without reading source code. This cannot be measured until a second user exists; it is a qualitative gate before the first public announcement.

## Caveats

1. **Metric 1 denominator may be incomplete.** The 4 occurrences in `src/docs/instructions.ts` are verified by code search. If Layer 3 template work (examples, new prompt files) re-introduces the string before the grep gate is live, the baseline count of 4 could be wrong at the time of measurement. **Fix:** add the grep as a CI check immediately with Layer 1, not after.

2. **Metric 2 has a second blind spot.** The instruction string check only verifies the text content emitted to the LLM. It cannot verify that a running pi agent does not invoke `mempalace_*` tools via an agent profile's `tools:` declaration. If the bundled `research-memory-scout.md` agent profile still lists `mempalace_search` in its tool declarations, the scout will attempt to call it even if the instruction text doesn't mention it. The smoke test must also grep agent profile YAML for `mempalace_` under the no-memory scenario.

3. **Bug C is hard to unit-test deterministically.** OSS scout alignment between `agents/research-oss-scout.md` tool declarations and `templates/prompts/scout-oss.md` prompt references is a structural consistency check. If the test asserts string equality between two files, it will break every time the prompt evolves. Better approach: parse both files and assert the declared tool names are a superset of the tool names mentioned in the prompt. If this is kept as a manual review item, the bug-fix test breadth target drops to 6/7.

4. **Metric 4 target count (тЙе5 tests) may shift.** The `ScoutDefinition` spec may surface additional edge cases during implementation (e.g., a scout that passes Stage 1 but times out in Stage 2). The 100% pass rate target is firm; the absolute count of 5 is a lower bound.

5. **Metric 5 scenario B boundary (exit 1 vs exit 0) depends on policy.** If the implementation treats a missing non-required provider as `exit 0` (warning in JSON only) rather than `exit 1`, the smoke test as written will fail scenario B. The smoke script should be written *after* the exit code policy is locked in the `research:doctor` implementation тАФ not before.

6. **The volunteer dogfood secondary signal assumes a second user exists.** If the only user during Layer 2 is the author, this cannot be measured objectively. Plan to solicit a single external tester before the first public release.
